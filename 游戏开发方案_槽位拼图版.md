# 《画栋飞甍》开发方案（槽位拼图版 · 2026）

> **依据**：[游戏设计更新.md](游戏设计更新.md)  
> **定位**：在时间与工程量约束下，将项目从「大系统网格建造」收敛为 **斜二测 + 异形槽位吸附拼图**，保留「像真在建房子」的部件语义与科普价值。

### 视觉与内容参考（目标成片）

- **参考**：视频《梁架篇…四架椽屋…》与 **构件顺序** 仅作参考；**关卡与科普文案以 [游戏设计更新.md](游戏设计更新.md) 为准**，不要求与视频逐帧对齐。
- **说明**：AI **无法直接播放 `.mp4`**；需要时以文档清单驱动制作即可。

### 吸附动画（已确认）

- 拼图片在 **有效吸附判定成立** 后，使用 **缓动平移** 滑入槽位目标点（默认 `SmoothStep` + `Lerp`，时长可在 `PuzzleDragRuntime` 调节），**禁止瞬移**；放错槽则 **缓动回到** 部件栏记录的 `HomeWorldPosition`。

## 一、我对更新设计的理解（是否正确）

| 维度 | 更新后的方向 |
|------|----------------|
| **核心玩法** | 以 **槽位（灰模/轮廓）** 为锚点；玩家拖动 **与槽形状匹配的拼图片（真实构件）**，靠近即 **吸附嵌入**，不是「把一张图切碎」的传统拼图。 |
| **空间** | **保留斜二测视角**；**取消「整建筑逻辑网格」** 作为玩法硬约束（不再依赖统一 cell 搭一切）。 |
| **与普通拼图差异** | **异形轮廓** + **放置先后顺序** + **遮挡/图层关系**（类似绘画图层），形成空间与阅读顺序上的谜题。 |
| **分工** | **程序/AI**：逻辑引擎、UI、流程、科普呈现、关卡构建辅助工具等（除关卡内容设计外尽量包圆）。**关卡作者**：主要 **摆槽位、设位置、设层级与遮挡**，必要时设顺序/依赖规则。 |
| **科普** | 玩家 **点击已放置（或槽位旁）的构件** → 弹出 **Galgame/VN 式「一句一句点下去」** 的短科普，避免长文墙。 |
| **UI** | **治愈、可爱** 的整体方向（色彩、圆角、动效、反馈音等）。 |
| **Demo 内容锚点** | 《营造法式》思路下 **「四架椽屋前后剳牵用四柱」** 厅堂造：按视频/文档给出的 **自下而上** 部件顺序，做成 **一条可玩的拼装关卡链**。 |
| **关卡设计即玩法 · 创意工坊（你新增）** | **把关卡设计直接做成游戏内能力**：懂行的玩家/作者先在 **编辑/参考模式** 用 **有色构件** 按正确营造顺序 **摆出一整套「参考摆法」**；工具再 **一键生成可发布关卡**：将各构件 **替换为同形、同位置的灰模槽位（Sprite 与拼图片相同，仅降低不透明度/去色）**，供其他玩家 **复现同一摆法**。该管线 **同时服务官方关卡设计** 与 **未来 UGC**。 |

若以上有任何一条与你的设想不一致，请直接指出；下文技术拆分会按上表执行。

## 二、新玩法的技术内核（可实现的「引擎」）

### 2.1 三类对象

1. **Piece（拼图片）**  
   - 有 **唯一构件 id**（如 `ludou_01`）、**展示 Sprite**、**碰撞/贴合用的形状定义**。  
   - **已拍板**：**不允许旋转**（仅固定朝向，降低校验与美术成本）。  
   - **形状与槽一致（已拍板）**：运行时 **不单独做「另一套几何」**；**槽位 Sprite = 与拼图片同一资源**，仅 **灰化 + 降低不透明度** 作为「待填灰模」。吸附校验用 **pieceId 匹配 + 位置/距离阈值** 即可（时间紧时优先简单方案）。  

2. **Slot（槽位）**  
   - **灰模表现**：与对应 **Piece 同 Sprite** 的 **灰度/低透明度** 版本（可在 Inspector 调 `Color` / `Material` / 第二张贴图）。  
   - **允许的 pieceId**、**排序层 Sorting Layer / Order**（遮挡）。  
   - **可选**：`requiredPreviousSlotIds`（必须先放谁才能激活本槽，体现先后）。  

3. **Board（关卡板）**  
   - 斜二测下的 **静态背景 + 若干 Slot 子物体 + 若干 Piece 生成点（料堆区）**。  
   - **关卡完成条件**：所有必选 Slot 已填入正确 Piece（或达到最小可演示集合）。

### 2.2 核心循环（玩家）

1. 从 **料堆/侧边栏 UI** 拖出一块 **Piece**（**有固定「出生点/栏位」**，用于失败回弹）。  
2. 靠近 **pieceId 匹配的 Slot** → **高亮提示** → **松开**后触发 **缓慢滑入**槽位动画（**无旋转**）。  
3. **放错槽**：**将 Piece 用缓动平移回** 拖出时的 **HomeWorldPosition**（与部件展示区一致）。  
4. 嵌入后 Piece **进入对应图层**，可能被后续构件 **部分遮挡**（符合真实搭建顺序）。  
5. **点击** 已嵌入 Piece → **科普对话** 逐句推进。  
6. 全部关键槽完成 → **结算/下一关**（Demo 可先一关到底）。

### 2.3 关卡编辑 → 可玩关卡（创意工坊管线 · 程序负责）

**目标**：**关卡设计 = 摆法 + 图层 + 导出**，与游玩时 **「灰槽 + 从 UI 取件」** 同源。

1. **参考模式（作者）**：在场景或专用编辑器中，用 **有色 Piece 预制体** 按 **建筑顺序** 摆放（与 `游戏设计更新.md` 及视频拆解 **同粒度**）。  
2. **导出命令**：`BakeLevelFromReference`  
   - 对每个已放置 Piece：记录 `pieceId + transform + sorting + order`。  
   - **生成** 对应 **Slot** 对象：同 transform、同 sorting、**Sprite 为同资源但灰模参数**；**隐藏或移除** 参考用有色 Piece（或移到另一 Layer 仅编辑可见）。  
   - **生成** 玩家侧 **料堆**：按关卡配置列出本关可用 `pieceId` 与数量（与视频同序拆件时，可与「当前可放批次」一致）。  
3. **玩家模式**：加载 **仅含 Slot + 背景 + UI**，玩家拖拽 **从 UI 取件** 复现摆法。  
4. **未来创意工坊**：同一套 `LevelData` 可 **打包分享**（JSON/Addressables/Steam Workshop 等，实现阶段再定 **最简文件交换**）。

### 2.4 与「旧网格建造」的本质区别

- 旧：在统一网格上「刷墙刷瓦」+ 复杂屋顶刷子。  
- 新：**每个构件是独立拼图块**，关卡难度来自 **轮廓匹配 + 顺序 + 图层**，工程量可控，且更贴「插槽式营造」视频演示。

## 三、系统模块划分（新版）

### A. 槽位拼图引擎（最高优先级）

| 子模块 | 功能 | 最小交付 |
|--------|------|----------|
| A1 | Piece 定义与生成 | ScriptableObject：`PieceDef`（id、sprite、**无旋转**） |
| A2 | Slot 定义与状态 | `SlotDef` / 场景组件：空/已填、目标 pieceId、sorting |
| A3 | 吸附与校验 | 距离+形状重叠（或 tag 匹配）+ 前置槽是否完成 |
| A4 | 图层与遮挡 | Sorting Layer / Order 规范；可选「后放的盖住先放的」规则表 |
| A5 | 关卡运行时 | `PuzzleLevelController`：开始、完成检测、失败提示（可选） |

### B. 输入与相机（沿用并收敛）

- **New Input System**：指针位置、按下/抬起、拖拽（与现有 `HfmInputRouter` 思路一致）。  
- **相机**：斜二测下 **平移 + 轻缩放**；**无网格吸附**。

### C. 科普子系统（VN 逐句）

| 子模块 | 功能 |
|--------|------|
| C1 | `LoreSnippet` 数据：多行文本、可选小图、关联 `pieceId` |
| C2 | `LorePopupUI`：点击下一句、跳过、关闭 |
| C3 | 点击 Piece 打开对应 Lore（未放置可点料堆预览，可选） |

### D. UI 框架（治愈可爱）

- **主菜单**、**关卡选择（Demo 可先 1 关）**、**HUD（进度/提示）**、**完成页**。  
- 视觉规范单独一页：`Art_UI_Guide.md`（色板、圆角、字体、按钮动效约定）。

### E. 关卡构建辅助（给你「只摆槽」的工具）

| 子模块 | 功能 |
|--------|------|
| E1 | **参考摆法编辑**：在 Scene 里用 **有色 Piece** 按真实顺序摆放（与视频/更新文档 **同粒度拆件**） |
| E2 | **一键烘焙**：有色参考 → **灰模 Slot** + `LevelData`（位置、图层、pieceId、依赖） |
| E3 | **导出/保存**：`LevelData`（ScriptableObject 或 JSON）：供运行时装载与后续 **创意工坊** 分发 |
| E4 | **校验报告**：缺引用、顺序矛盾、未覆盖必选槽等 |

> 目标：**关卡设计者几乎不写代码**，**先摆出一套「标准答案」**，再 **自动生成** 玩家面对的 **灰槽关卡**。

### F. Demo 关卡内容（设计向）

- **构件粒度（已拍板）**：与 [游戏设计更新.md](游戏设计更新.md) 中的分层清单 **一致地细**；建造顺序以该文 + 参考视频 **《梁架篇…四架椽屋…》** 为准（自下而上、铺作堆叠顺序等）。  
- **关卡切片顺序**：地基与柱 → 铺作（斗拱堆叠）→ 梁架（剳牵、四椽栿、蜀柱叉手、平梁）→ 屋面（檩、椽、望板）。  
- 每一 **真实构件名** 对应 **至少一个 Piece + 一个同形灰模 Slot**；若某一步在视频中为 **组合体**，可 **拆成多个子构件 Piece**（仍保持与文档同级粒度）。

### G. 明确降级 / 延后（除非你又改回大系统）

- **整建筑自由网格城镇**：不作为 Demo 必达；已有城镇原型可 **冷冻** 或仅作彩蛋。  
- **屋顶刷子连续铺设**：改为 **离散大块 Piece + 槽位**（更符合新设计）。  
- **知识卡片大图鉴**：可被 **VN 逐句 Lore** 部分替代；后续再合并为图鉴。

## 四、与现有工程的关系（复用 vs 重构）

### 建议复用（改语义即可）

- `HfmInputRouter`、拖拽逻辑 → 接到 **Piece 拖动** 上。  
- `SnapSlot` / `SnapManager` 思想 → 演进为 **Slot + 形状/规则匹配**，不必再依赖 `GridSystem` 主循环。  
- `SaveSystem` → 存关卡进度、解锁；Demo 可先只做 **PlayerPrefs / 单 JSON**。  

### 建议逐步替换或旁路

- `GridSystem` 作为 **全局玩法** → **弃用**；若需要仅用于 **编辑器对齐辅助** 可保留为 Editor-only。  
- `RoofLevelController` 刷子式屋顶 → **Demo 不主用**；逻辑可参考「连续生成」改为「一次生成一排椽条 Piece 预制体」若仍要表现铺椽。  
- `DouGongLevelController` 基于距离检测的规则 → 合并进 **统一 Puzzle 引擎**，避免两套校验。  

### 可推翻重来（若工期更紧）

- 直接新建 `Assets/Scripts/Puzzle/` 命名空间，旧 `Building/` 保留在场景外 **Legacy** 文件夹，避免混淆。

## 五、开发阶段规划（建议顺序）

### 阶段 P0：方案冻结与资源约定（1–3 天）

- 冻结：**PieceDef / Slot 烘焙规则**、**图层命名**、**灰模材质/透明度参数**、**料堆 UI 布局**。  
- **构件列表**：以 `游戏设计更新.md` 全文为 checklist，按视频顺序 **能做多细做多细**；工程上 **怎么简单怎么实现**（同 Sprite 灰模 + id 匹配优先）。

### 阶段 P1：垂直切片「单根柱 + 单栌斗」（3–7 天）

- 1 个背景 + 2 个槽 + 2 块拼图 + 吸附 + 完成判定 + 一次 Lore 点击。  
- 验证：斜二测下图层遮挡是否好看、拖拽手感是否解压。

### 阶段 P2：Demo 关卡「四架椽屋」骨架（按文档顺序分段）（2–4 周）

- 按 **地基→柱→阑额→斗拱层→梁架→屋面** 分段提交；每段可玩、可点科普。  
- 同步：**关卡编辑辅助** 最小版（能保存槽位列表）。

### 阶段 P3：UI 与包装（与 P2 重叠）

- 主菜单、加载、设置（音量）、完成页；**治愈可爱** 一轮迭代。

### 阶段 P4：打磨与扩展入口

- 音效、粒子、成就（可选）、第二关模板；考虑是否恢复「城镇」为独立模式。

## 六、Demo 关卡：部件 → 关卡任务映射（初稿）

> 用于和美术/策划对表；具体「一块还是多块拼图」可按贴图拆分再定。

1. **地基与支撑**：台基地栿、柱础、檐柱/内柱、阑额与由额 → 各对应 Slot + Piece（先下层后上层）。  
2. **铺作层**：栌斗 → 华拱/泥道拱 → 交互斗/齐心斗/散斗 → 令拱 → 严格顺序 + 图层。  
3. **梁架层**：剳牵、四椽栿、蜀柱与叉手、平梁。  
4. **屋面层**：脊檩/平檩/檐檩、檐椽与飞椽、望板。  

**科普文案**：每个 **PieceDef** 挂 `LoreSnippet` 或引用 id，由策划填句，程序只负责播 UI。

## 七、已拍板事项（2026-03-16）

| 项 | 结论 |
|----|------|
| **旋转** | **不允许**；拼图片仅固定朝向。 |
| **放错槽** | **平移回到** 从 **部件展示 UI** 拖出时的 **原位置/栏位**（不做复杂震动为主）。 |
| **构件拆细** | 与 **游戏设计更新.md** 一致，并参考 **《梁架篇…四架椽屋…》** 视频建造顺序；**能多细做多细**。 |
| **槽与拼图** | **同一张 Sprite**；槽为 **拼图片降低透明度 + 灰模**（视觉一致、实现最省）。 |
| **工期策略** | **怎么简单怎么来**；优先 **id 匹配 + 位置吸附 + 排序图层**，避免过度几何库。 |
| **语言** | 默认 **仅中文**；若后续要本地化再抽 key（不阻塞首版）。 |

### 仍可选补充（不阻塞开发）

- 是否需要在 **放错** 时加 **一句极短可爱文案**（仍配合平移回 UI）。  
- 创意工坊 **第一版** 仅 **本地导出/导入关卡文件**，还是只要 **内置关卡列表**。
